## 実装戦略 (C# WPF)

### オペコードディスパッチ

65536 エントリのデリゲートテーブル (`Action<ushort>[65536]`)。各エントリは
InstructionDecoder のメソッドへのデリゲート。`InitOpcodeTable()` で初期化し、
ホットオペコードは専用 fast handler で上書き登録。

### Fast ハンドラ

高頻度オペコードは Group デコーダをバイパスして直接ディスパッチ:

| ハンドラ | オペコード | 説明 |
|---------|---------|------|
| FastMOVEQ | 0x7000-0x7FFE (全 Group7) | ムーブクイック即値。ハンドラ内で bit8 チェック、不正時は例外4 |
| FastMOVE_L_Dn_Dm | MOVE.L Dn,Dm | レジスタ間ロングワード転送 |
| FastBRA_B | 0x6001-0x60FE | 8ビットディスプレースメント付き無条件分岐 |
| FastBcc_B | 0x6x01-0x6xFE (cond!=0) | 8ビットディスプレースメント付き条件分岐 |
| FastRTS | 0x4E75 | サブルーチンからの復帰 |
| FastADD_L | ADD.L Dn,Dm | レジスタ間ロングワード加算 |
| FastSUB_L | SUB.L Dn,Dm | レジスタ間ロングワード減算 |
| FastCMP_L | CMP.L Dn,Dm | レジスタ間ロングワード比較 |

C++ 版との違い: C# では FastBRA_B を専用ハンドラとして登録（C++ は DecodeGroup6 で処理）。
FastMOVEQ は全 Group7 オペコードに登録し、ハンドラ内で bit8 ガード（C++ はテーブル
登録時にガード）。

### レジスタスナップショット

遅延レジスタスナップショットは未採用。C++ では有効だが、C# では JIT オーバーヘッドに
より速度低下したためリバート。各命令実行前に都度スナップショットを取得。

### JIT コンパイラ

System.Reflection.Emit による IL コード生成。レジスタ専用命令ブロックを .NET IL に
コンパイルし、DynamicMethod として実行。

- ファイル: `Core/Jit/CompiledBlock.cs`, `Core/Jit/JitCache.cs`, `Core/Jit/JitCompiler.cs`
- ExecuteNextFast() と ExecuteNextFastJit() は完全に別メソッド（C++ と同様の分離設計）
- bool 分岐でメソッド選択。Func<bool> デリゲートディスパッチは 36→30 MHz の
  回帰を引き起こしたため不採用。
- JIT ON は現時点で JIT OFF より遅い (31.0 vs 32.1 MIPS): ExecuteNextJit の NoInlining
  メソッド呼出オーバーヘッド + block.Execute() の DynamicMethod デリゲートディスパッチ
  コストが、コンパイル可能ブロックの希少さを上回る。

### FPU 診断トレース

FpuInstructionDecoder に `FpuTrace()` メソッドあり。`FpuTraceEnabled` フラグで
FPU 命令のトレース出力が可能（C++ 版にはない機能）。

### 性能 (JIT OFF)

約32.14 MIPS / 約216.75 MHz サイクル (Avg モード実測)、約6.7 サイクル/命令

> **注意**: これらの数値はサイクル精度の計測値ではなく、概算の推定値です。
> - **MHz** = エミュレート済みサイクル総数 / 実時間（秒） / 1,000,000。サイクル数は
>   65,536 エントリの静的ルックアップテーブル (`s_cycleTable`) と EA コスト調整から
>   算出しており、MC68030 のタイミングを近似しますがパイプライン・キャッシュ・
>   バスウェイトステートはモデル化していません。
> - **MIPS** = エミュレート済み命令総数 / 実時間（秒） / 1,000,000。
> - **Avg** 値は Run セッション開始からの累積値。瞬時値は約 500ms 間隔でサンプリング。
> - 結果はワークロード、ホスト CPU、システム負荷により変動します。

> **Linux 実行時に MHz/MIPS が NetBSD より極端に低くなる現象について**: Linux カーネル
> 実行時、表示される MHz および MIPS の値が NetBSD と比較して大幅に低くなることが
> あります。この現象はアイドル状態のシェルプロンプト待機中に最も顕著で（例: 約 1.5 MHz
> / 約 0.2 MIPS）、ブート中にも確認できます。主に以下の2つの要因によるものです。
>
> 1. **STOP 命令によるアイドル時間** — Linux は割り込み待ちに M68K の `STOP` 命令
>    (0x4E72) を使用して CPU を停止させます（割り込み駆動型アイドル）。STOP 中、
>    エミュレータは命令を実行せずサイクルも加算しませんが、実時間は経過し続けます。
>    MHz と MIPS はサイクル数（または命令数）を実時間で除算して算出するため、STOP
>    期間が長いと両指標が大きく低下します。シェルプロンプト待機中はほぼ全時間が
>    STOP 状態のため、値は極端に低くなります。一方 NetBSD はビジーウェイトループで
>    命令を連続実行する傾向があり、アイドル時でもカウンタが高い値を維持します。
>
> 2. **SCSI 割り込み処理の不完全さ** — WD33C93 SCSI コントローラのエミュレーションが、
>    Linux カーネルドライバが期待するすべての割り込みシナリオに対応していません。
>    SCSI ディスクプローブ中、未処理の割り込み条件によりドライバがポーリングループに
>    入ったり、到着しないイベントを待ち続けたりするため、実効的な命令スループットが
>    さらに低下します。これは WD33C93 の割り込みエミュレーション改善により対処予定の
>    既知の制限事項です。
>
> 低い値は MHz/MIPS 計測自体のバグを示すものではなく、エミュレートされた CPU が
> 大部分の時間をアイドルまたはストール状態で過ごしていることを正確に反映しています。

### サイクルテーブル

アドレッシングモードごとのメモリアクセスオーバーヘッドを計算。

## JIT まとめ

基本ブロック単位で命令を JIT コンパイル。レジスタ専用ブロックは完全に JIT 内で実行。
メモリアクセスブロックは**データページキャッシュベイルアウト**機構を使用:
キャッシュヒット時は JIT 内で継続、キャッシュミス時は当該命令の PC で
インタプリタにベイルアウト。ベイルアウト頻度が高いブロック（>64回）は
ブラックリスト化して除外。

約60種の命令パターンに対応:

| カテゴリ | 命令 |
|---------|------|
| 転送（レジスタ） | MOVEQ, MOVE.L Dn->Dm, MOVE.L An->Dn, MOVEA.L Dn->An, MOVEA.L An->Am |
| 転送（メモリ） | MOVE.L (An)->Dm, (An)+->Dm, Dm->(An), d16(An)->Dm, Dm->d16(An)（ベイルアウト） |
| 算術 | ADD/SUB/CMP .B/.W/.L (Dn,Dm), ADDQ/SUBQ .B/.W/.L (Dn, An), NEG .B/.W/.L Dn |
| 乗算 | MULU.W, MULS.W (Dn,Dm) |
| 論理 | AND/OR/EOR .B/.W/.L (Dn,Dm), NOT/CLR/TST .B/.W/.L Dn |
| シフト | ASL/ASR/LSL/LSR.L #imm,Dn |
| ビット | BTST Dn,Dm |
| レジスタ | EXG（全3形式）, SWAP Dn, EXT.W/EXT.L/EXTB.L Dn |
| アドレス | LEA (An)/d16(An)/d8(An,Xn),Ar |
| 分岐 | BRA .B/.W, Bcc .B/.W（全条件） |
| サブルーチン | RTS（ベイルアウト） |
| その他 | NOP |

### C# JIT 実装詳細

- System.Reflection.Emit による IL コード生成 (DynamicMethod)
- コンパイル時に各オペコードのサイクルコストを合算し CompiledBlock.TotalCycles に格納
- サンプリング間隔 TickInterval (256)、コンパイル閾値 = 16 ヒット

#### チューニングパラメータ

以下のパラメータは 設定 → パフォーマンス → JIT で変更可能:

- **Compile Threshold**（デフォルト: 16）: 基本ブロックがコンパイルされるまでに必要な
  サンプリング検出回数。エミュレータは TickInterval (256) 命令ごとに現在の PC を
  サンプリングし、同一ブロックアドレスがこの回数に達するとコンパイルが実行される。
  値を小さくするとより多くのブロックが早期にコンパイルされるが、コンパイルの
  オーバーヘッドも増加する。値を大きくすると最もホットなループのみがコンパイル対象になる。
  現時点では JIT は全体的にマイナスの性能影響（MIPS で約 3.5% 低速）があるため、この値の
  調整による実用的な効果は限定的。

- **Min Block Length**（デフォルト: 3）: キャッシュ対象とする基本ブロックに必要な
  JIT コンパイル可能命令の最小数。この閾値未満のブロックはコンパイル不可として即座に
  マークされ、頻繁に実行されてもコンパイルされることはない。これにより、ブロックごとの
  セットアップコスト（ルックアップ、デリゲート呼び出し）が利益を上回る極めて短い
  ブロックのコンパイル・ディスパッチオーバーヘッドを回避する。値を大きくすると
  コンパイルされるブロック数が減少し、小さくするとより多くのブロックがコンパイルされるが、
  微小なブロックに対するディスパッチオーバーヘッドが増加する可能性がある。
- ExecuteNextFast() と ExecuteNextFastJit() は完全分離（C++ と同様の設計）
- メソッド選択は bool 分岐。Func<bool> デリゲートディスパッチは 36→30 MIPS 回帰を
  引き起こしたため不採用（定方向分岐は分岐予測によりほぼゼロコスト）
- ベイルアウト機構: メモリアクセス命令は `TryReadLongCached()` を呼び出し、
  `_jitExecutedCount`/`_jitExecutedCycles` サイドチャネルフィールドで部分実行を通知。
  64回超のベイルアウトでブラックリスト化。
- テスト: 327 件（JIT 専用テスト含む、Em68030.Tests/）

### 性能

| モード | MHz（サイクル） | MIPS | 備考 |
|--------|----------------|------|------|
| JIT OFF | ~217 | ~32.1 | ベースライン |
| JIT ON | ~190 | ~31.0 | -3.5% MIPS / -12.3% MHz オーバーヘッド |

JIT ON は現時点で JIT OFF より遅い（-3.5% MIPS、-12.3% MHz）。ExecuteNextJit の
NoInlining メソッド呼出オーバーヘッド + block.Execute() の DynamicMethod デリゲート
ディスパッチコストが、コンパイル可能ブロックの希少さを上回る。
JIT はデフォルトで無効。
